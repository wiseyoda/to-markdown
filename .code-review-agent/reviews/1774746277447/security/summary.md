## Security Review Summary

Read 30+ source files across `src/to_markdown/` (CLI, core pipeline, MCP server, smart/LLM modules, background task system, SQLite task store, content sanitization, batch processing, setup wizard). Traced all data flows from user input (CLI args, MCP tool parameters, file content, Gemini API responses) through to dangerous sinks (filesystem I/O, SQL queries, subprocess spawning, LLM prompt construction).

No actionable security findings. The codebase follows sound security practices: SQL queries use parameterized values, subprocess calls use list args (no `shell=True`), no `eval()`/`exec()`/`pickle.loads()`/`yaml.load()` on untrusted data, API keys come from environment variables, and content sanitization strips non-visible Unicode characters. The `TaskStore.update()` method dynamically builds SQL SET clauses from Python kwargs, but all callers use hardcoded key names with no path for user input to reach column names. The `.format()` calls in LLM prompt construction use constant templates with extracted content as values, posing no format string injection risk. The MCP server uses stdio transport with no network listener, limiting the attack surface to local filesystem operations.
