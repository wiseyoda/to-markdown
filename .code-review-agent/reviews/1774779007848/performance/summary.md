# Performance Review Summary

Reviewed 20 source files across the conversion pipeline (cli.py, pipeline.py, content_builder.py, batch.py, sanitize.py, extraction.py, frontmatter.py, tasks.py, worker.py, background.py, display.py, cli_helpers.py, setup.py, constants.py, mcp/tools.py, mcp/background_tools.py, smart/llm.py, smart/clean.py, smart/images.py, smart/summary.py). Analyzed hot paths for algorithmic complexity, I/O amplification, resource issues, and unnecessary work. Found 2 findings: (1) `sanitize_content` uses a Python-level character loop where `str.translate()` would be 5-10x faster on large documents (medium), and (2) CLI batch mode processes files sequentially while the async variant already supports inter-file parallelism via `asyncio.gather()` (low). The codebase is otherwise well-structured for performance — LLM calls use async parallelism within files, Kreuzberg extraction is delegated to Rust, the SQLite task store uses WAL mode, and the Gemini client is properly cached as a singleton.
