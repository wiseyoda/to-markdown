## Performance (2)

- [ ] **[CRITICAL]** Synchronous file extraction blocks the async event loop <!-- id:92f08618-d3dd-4f2c-8527-951610df0fcd -->
  - **File:** `src/to_markdown/core/content_builder.py:51`
  - The `build_content_async` function is designed for use inside a running async event loop (like the FastMCP server). However, it calls the synchronous `extract_file` function, which performs blocking disk I/O and CPU-intensive Kreuzberg document extraction on the main thread.

  - **Fix:** Use asyncio.to_thread to run the synchronous extraction in a worker thread: `result = await asyncio.to_thread(extract_file, input_path, extract_images=images)`


- [ ] **[HIGH]** Sequential processing in convert_batch_async limits throughput <!-- id:84e11f77-d430-4949-85c8-46cf940afde6 -->
  - **File:** `src/to_markdown/core/batch.py:180`
  - In convert_batch_async, files are processed sequentially using a for loop with `await convert_file_async()`. Since convert_file_async involves multiple network calls to the LLM API for each file, doing this sequentially prevents taking advantage of parallel network I/O.

  - **Fix:** Process the files concurrently using asyncio.gather and an asyncio.Semaphore to limit concurrency.


## Code Quality (3)

- [ ] **[HIGH]** Background batch worker breaks glob-based conversions <!-- id:a9018918-19fc-43ff-ad53-31021909557a -->
  - **File:** `src/to_markdown/core/worker.py:99`
  - `run_worker()` treats every background batch job as a directory walk by converting `input_path` to `Path` and passing it to `discover_files()`. The CLI marks glob patterns such as `docs/*.pdf` as `is_batch`, so a background job created from a glob reaches this branch with a non-directory path literal and never resolves the pattern into matching files.

  - **Fix:** Persist whether the batch input was a directory or a glob when the task is created, then branch here accordingly. For glob jobs, call `resolve_glob(input_path)` and fail the task if it returns no files instead of sending the pattern through `discover_files()`.


- [ ] **[MEDIUM]** Background batch tasks hide per-file failures <!-- id:ff7016e1-406b-4890-8b1a-f95f4711c12b -->
  - **File:** `src/to_markdown/core/worker.py:115`
  - After `convert_batch()` returns, `run_worker()` always stores `TASK_STATUS_COMPLETED` for the task regardless of whether `result.failed` contains conversion errors. The only persisted output is a summary string like `"1 succeeded, 2 failed"`, so the task state itself no longer reflects that the batch encountered failures.

  - **Fix:** Set the stored status from the batch result instead of hard-coding `TASK_STATUS_COMPLETED`. At minimum, mark the task failed when `result.failed` is non-empty and include the failed file details in the stored error field; a dedicated partial-success status would be even clearer if the task model can be extended.


- [ ] **[MEDIUM]** spawn_worker leaks a log file descriptor in long-lived processes <!-- id:b56aab72-0344-4e82-a85a-cd0d05a56850 -->
  - **File:** `src/to_markdown/core/worker.py:35`
  - `spawn_worker()` opens the per-task log file in the parent process and passes that handle into `subprocess.Popen()`, but it never closes the parent-side descriptor after the child has been launched.

  - **Fix:** Close `log_fd` in the parent after `Popen()` succeeds, ideally with a `try/finally` around the spawn so the descriptor is also closed on launch failures. The child can keep using the inherited descriptor for stdout and stderr.

  - **File:** `src/to_markdown/core/worker.py:100`
  - The background batch worker path always calls `discover_files(source)` with its default recursive behavior when `is_batch` is set, and the CLI background command only serializes `force` and `sanitize` into the task payload. I could not find any test that starts a background directory conversion with `--no-recursive` and then verifies either the stored command arguments or the worker's call to `discover_files(..., recursive=False)`, so this branch can regress without the suite noticing.

  - **Fix:** Add an integration-style test that invokes the CLI with a directory, `--background`, and `--no-recursive`, then asserts the queued task payload preserves the recursion choice and `run_worker()` calls `discover_files(source, recursive=False)`.
